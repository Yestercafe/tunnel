# Features Research

**Domain:** Session 中继隧道（广播 + 私信、双向流）  
**Researched:** 2026-03-29  
**Confidence:** MEDIUM

## Table Stakes（v1 应覆盖或明确排除）

| 能力 | 说明 | 复杂度 |
|------|------|--------|
| 创建 session / 加入 session | 创建者得 id 或邀请码；peer 凭此加入 | 中 |
| 同 session 广播 | 除发送者外均收到 | 中 |
| 私信（单播） | 指定目标 peer | 中 |
| 双向流 | 会话内可持续读写 | 中高 |
| 小消息 + 大块流 | 分片、流控、背压需在规范中占位 | 高 |
| 流内有序、流间乱序 | 用流 ID 区分顺序域 | 中 |
| 可选应用信封 | content-type、请求 id、关联 id | 低中 |
| TLS 字节流上成帧（典型 TCP） | v1 主路径；与进程/服务 client 对齐 | 中 |
| 协议版本与能力协商 | 帧头或握手阶段 | 中 |

## Differentiators（可后续版本）

| 能力 | 说明 |
|------|------|
| 端到端加密 | 与当前「TLS 在边缘」假设不同，需单独设计 |
| 大规模集群与分片 | 当前假设少量固定 peer |
| 丰富鉴权（OAuth 等） | v1 可用短 token，完整 IAM 后置 |

## Anti-Features（刻意不做或推迟）

| 能力 | 原因 |
|------|------|
| 在 v1 强制完整服务端产品化 | 与「先规范 + 一致性测试」冲突 |
| 隐式全局全序 | 与「流间乱序」决策冲突；若需要全局序应显式新增语义 |

## Dependencies Between Features

- **广播/私信** 依赖 **成员标识** 与 **路由表** 语义。  
- **大块流** 依赖 **流 ID** 与 **分片**；在 **TLS 字节流**上需写清**帧边界**与分片重组。  
- **应用信封** 可与帧头正交：先解析帧，再解析可选信封。

## Feature Complexity Notes

- **私信**：需在帧或信封层携带 `to` 或等价路由字段。  
- **浏览器（v2+）**：若增加 WebSocket/WebTransport 适配，再单独写与逻辑帧的映射；**v1 不强制**。
