# 安全假设与信任边界（SEC-01）

## 范围

本文档描述 **Tunnel v1** 在 **传输与协议**层面的**非形式化**安全假设与**明确不提供的保证**。实现者与部署方应结合 **SEC-01**（见 `.planning/milestones/v1.0-REQUIREMENTS.md` 追溯表）与本文一并阅读。

## 信任边界：TLS 在边缘 / 服务器终止

- **客户端到 TLS 终结点**：机密性与完整性主要由 **TLS** 与 **服务端/边缘证书所标识的身份域** 提供（具体证书校验策略由部署决定，不在 v1 帧格式中编码）。
- **TLS 解密之后**：字节流进入 **Relay（或等价服务端）进程**。在该进程内，协议实现将数据解析为 **[frame-layout.md](./frame-layout.md)** 定义的 **逻辑帧**；此时载荷对 **Relay 软件栈** 而言为 **明文**（除非部署另行叠加加密，见下文「不提供」）。

## v1 明确不提供

| 不提供的保证 | 说明 |
|--------------|------|
| **端到端加密（E2E）** | v1 **不在** peer 与 peer 之间提供独立于 Relay 的端到端机密性；**防中继窃听**需 **部署隔离**、**mTLS**、**应用层加密** 或 规划中的 **v2 SEC-02**（需求追溯见 `.planning/milestones/` 下归档 `*-REQUIREMENTS.md`）。 |
| **中继不可见内容** | **Relay 进程**若按协议解析或转发，则 **可观测** 帧级与 payload 边界；产品若要求「中继不可读」，须在 **协议外** 解决。 |
| **形式化验证** | 本文**不是**对 TLS、PKI 或实现的证明；仅列工程假设。 |

## 威胁与缓解（短表）

| 威胁 / 风险 | 缓解或接受策略 |
|-------------|------------------|
| **中继管理员或同机进程读取明文帧** | 接受为 v1 信任模型的一部分；敏感负载用 **应用层加密** 或 **网络隔离**；长期需求见 **SEC-02**。 |
| **证书误配、主机名校验缺失** | 部署 **严格 TLS 校验**、监控与轮换；协议层仅假设 TLS 已建立。 |
| **日志与调试导出明文** | 运维策略：**限制** payload 日志、脱敏；与 **ERR-01** 错误路径分开治理。 |
| **TLS 会话与 TCP 被旁路或降级** | 依赖 **部署**（仅 TLS 入口、HSTS/等价策略 等），非 v1 帧内可解。 |

## 推荐部署方向（非规范性）

- **同信任域内** 多 Relay：优先 **网络隔离** 与 **身份到实例的 mTLS**，缩小明文暴露面。
- **跨公网 peer**：至少 **TLS 到边缘**；若内容须对中继保密，在 **v2** 跟踪 **SEC-02** 或在应用层加密后再封装进 **`application_data`**。
